# 微信接入开发规划

## 项目目标

让「灵伴」AI 智能体能够通过微信与用户进行正常聊天：

- 用户可以在微信里直接给智能体发消息；
- 智能体基于已保存的人设、模型、API Key 回复；
- 多轮对话与网页端数据互通；
- 支持文字、图片等常见消息类型；
- 后续持续优化体验与稳定性。

---

## 阶段一：基础准备与架构设计

**目标**：确定接入形态，完成账号与基础设施准备。

1. 选择接入方式
   - 公众号（订阅号/服务号）：适合被动接收消息、主动客服消息，用户无需安装 App；
   - 微信小程序：适合富交互、语音/图片、自定义 UI；
   - 企业微信：适合 B 端、内部办公场景；
   - 个人微信机器人（Hook/PC 协议）：风险高、不稳定，暂不建议。

   **推荐**：先实现微信公众号（服务号）接入，再扩展到小程序。

2. 注册与配置
   - 注册微信公众号/小程序，完成认证；
   - 获取 AppID、AppSecret；
   - 配置服务器 URL、Token、EncodingAESKey（可选消息加解密）。

3. 技术架构设计
   - 新增 `WechatModule`：
     - `wechat.controller.ts`：接收/校验微信服务器推送；
     - `wechat.service.ts`：消息解析、回复组装、调用 LLM；
     - `wechat-user.service.ts`：OpenID 与本地用户绑定。
   - 数据模型扩展：
     - `WechatUser`：openid、unionid、nickname、avatar、boundUserId；
     - `Agent` 增加 `wechatEnabled` 字段，控制是否允许微信访问；
     - `ChatMessage` 增加 `source` 字段（web/wechat）。

4. 安全设计
   - 微信签名验证；
   - 消息加解密（Mode 2/3）；
   - AccessToken 集中缓存与刷新。

---

## 阶段二：微信服务器对接

**目标**：完成微信服务器验证与消息收发通道。

1. 服务器 URL 接口
   - `GET /api/wechat/:agentId`：
     - 解析 signature、timestamp、nonce、echostr；
     - 按微信规则校验 Token；
     - 返回 echostr 完成接入。

2. 接收消息接口
   - `POST /api/wechat/:agentId`：
     - 校验签名；
     - 解析 XML 消息（文本、图片、语音、事件等）；
     - 解密消息（如启用安全模式）；
     - 提取 fromUserName（OpenID）、msgType、content。

3. 被动回复消息
   - 5 秒内返回 XML 响应（文本/图片）；
   - 若 LLM 响应慢，先回复「正在思考…」，再通过客服消息异步推送。

4. 主动客服消息
   - 获取并缓存 `access_token`；
   - 调用微信客服消息接口 `POST https://api.weixin.qq.com/cgi-bin/message/custom/send`；
   - 处理 48 小时互动限制、异步队列与重试。

---

## 阶段三：用户体系打通

**目标**：让微信用户与网页端匿名用户数据互通。

1. OpenID 映射本地用户
   - 首次收到消息时，创建匿名 `User` 并绑定 `WechatUser`；
   - 后续根据 OpenID 找到对应用户和会话。

2. 网页端绑定微信
   - 生成带 state 的二维码（公众号场景）或小程序码；
   - 用户扫码/授权后，将微信 OpenID 与当前网页 `User` 绑定；
   - 绑定后两端消息同步。

3. 多设备会话同步
   - 同一用户的 `ChatMessage` 不区分来源，统一按 `agentId + userId` 查询；
   - `source` 字段仅用于展示来源标识。

---

## 阶段四：智能体对话接入

**目标**：微信消息接入现有 LLM 对话流程。

1. 消息路由
   - 根据 `agentId`（URL 参数或用户默认智能体）确定目标智能体；
   - 若用户未指定，使用最近对话的智能体或默认智能体。

2. 复用现有 LLM 服务
   - 将微信消息转换为 `ChatMessage`（role=user）；
   - 读取历史记录组装 messages；
   - 调用现有 SSE/流式接口或非流式接口获取回复。

3. 微信侧回复
   - 文本：直接返回或主动推送；
   - 长文本：拆分成多条消息；
   - 图片/语音：先生成 URL，再组装为微信媒体消息（需先上传素材）。

4. 上下文管理
   - 复用数据库历史记录；
   - 微信 48 小时限制下，超过时间可提示用户重新发起。

---

## 阶段五：功能完善与优化

**目标**：提升稳定性、扩展消息类型、优化体验。

1. 消息类型扩展
   - 图片消息：接收图片后做 OCR 或图文理解；
   - 语音消息：语音识别（STT）后作为文本输入；
   - 事件消息：关注/取消关注、菜单点击；
   - 模板消息：用于通知、任务完成提醒。

2. 性能与稳定性
   - 微信消息队列（Redis/BullMQ），削峰填谷；
   - LLM 超时兜底，避免微信重试导致重复回复；
   - 幂等控制：根据微信 MsgId 去重。

3. 多智能体管理
   - 每个智能体独立微信入口 URL；
   - 用户在网页端设置「允许微信访问」开关；
   - 支持通过菜单/关键词切换智能体。

4. 监控与日志
   - 记录微信消息日志；
   - 接入 Sentry/Loki 告警；
   - 统计消息量、响应时间、错误率。

---

## 阶段六：测试与上线

**目标**：安全、稳定地发布到生产环境。

1. 沙箱测试
   - 使用微信测试号验证消息收发；
   - 测试签名、加解密、access_token 刷新；
   - 模拟超时、重试、异常消息。

2. 灰度发布
   - 先对一个智能体开启微信入口；
   - 观察 Render 日志与微信告警；
   - 逐步全量。

3. 上线检查清单
   - 服务器 URL 可公网访问（Render HTTPS）；
   - 消息加解密配置正确；
   - AccessToken 缓存正常；
   - 数据库迁移已应用；
   - 回滚方案就绪。

## 风险与注意事项

1. 微信规则限制
   - 公众号被动回复 5 秒超时；
   - 客服消息需在用户最后一条消息 48 小时内；
   - 内容安全：需接入微信文本/图片内容安全接口。

2. 合规
   - 用户数据与隐私政策；
   - LLM 生成内容的免责声明。

3. 成本
   - 微信公众号认证费用；
   - LLM API 调用成本随消息量增长；
   - Render 服务与数据库费用。
