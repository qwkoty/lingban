# 灵伴 AI 智能体开发文档

> 版本：v1.0  
> 目标：将当前基础可用的网页版智能体应用，按三阶段打磨为一个像 Character.AI / 角色陪伴类产品那样的「AI 好友」体验——智能体拥有鲜明性格、能主动发消息、记住用户偏好，并最终部署在 Render 上，同时接入微信。

---

## 1. 项目现状

### 1.1 产品定位

「灵伴」不是传统意义上的工具型 AI 助手，而是一个**AI 好友/角色陪伴**应用。每个智能体都像用户拥有的一位虚拟朋友：有自己鲜明的性格、背景、说话方式，会主动关心用户、找话题聊天，而不是只会被动回答问题。用户无需注册账号即可匿名使用，可创建多个不同性格的 AI 好友，在移动端随时聊天。产品以移动端优先，采用极光渐变 + 毛玻璃拟态风格。

### 1.2 已完成功能

| 模块 | 已支持 | 关键文件 |
|---|---|---|
| 匿名登录 | 首次访问自动创建用户，Token 持久化 | `api/routes/auth.ts`、`src/store/auth.ts` |
| 用户资料 | 昵称、头像、人设、主题色 | `src/pages/ProfilePage.tsx` |
| 智能体管理 | 增删改查、头像上传、性格人设/模型/API Key 配置 | `src/pages/AgentEditPage.tsx`、`api/routes/agents.ts` |
| 多轮对话 | 流式 SSE 回复、历史记录持久化 | `src/pages/ChatPage.tsx`、`api/routes/chat.ts` |
| 主题切换 | 极光 / 七彩两套主题 | `src/store/theme.ts` |
| 部署目标 | Render Blueprint 已配置，作为默认生产环境 | `render.yaml` |

### 1.3 技术栈

- **前端**：React 18 + TypeScript + Vite + Tailwind CSS + Zustand + React Router
- **后端**：Express 4 + TypeScript（ESM）
- **数据库**：PostgreSQL + Prisma ORM
- **LLM**：OpenAI 兼容格式，当前代码硬编码 DeepSeek 端点

### 1.4 项目结构

```text
/workspace
├── src/               # React 前端
│   ├── components/    # 通用组件
│   ├── pages/         # 页面
│   ├── store/         # Zustand 状态
│   ├── lib/           # API 封装
│   └── App.tsx        # 路由入口
├── api/               # Express 后端
│   ├── routes/        # 业务路由
│   ├── middleware/    # 鉴权中间件
│   └── app.ts         # Express 应用入口
├── prisma/            # 数据库 Schema 与迁移
├── public/            # 静态资源
└── package.json
```

### 1.5 数据模型

核心表：`users`、`agents`、`chat_messages`。

关键字段：

- `users.token`：匿名用户凭证
- `users.persona` / `agents.persona`：人设/系统提示词
- `agents.modelProvider` / `agents.modelName` / `agents.apiKey`：模型配置
- `chat_messages.role` / `content`：消息角色与内容

完整定义见：`prisma/schema.prisma`

### 1.6 当前已知缺口

1. **模型提供商未生效**：`AgentEditPage` 可选择 OpenAI/Anthropic/DeepSeek/自定义，但 `api/routes/chat.ts` 仍固定调用 `https://api.deepseek.com/chat/completions`。
2. **缺少 AI 好友核心体验**：智能体不会主动发消息，没有开场白、主动关怀或找话题的能力。
3. **缺少长期记忆**：无法记住用户的关键信息（名字、喜好、最近聊过什么），导致对话缺乏连续性。
4. **缺少会话概念**：同一智能体只有一条连续历史，不支持多组独立会话。
5. **对话交互不完善**：没有重新生成、清空对话、消息重试、编辑消息、复制消息等功能。
6. **错误提示薄弱**：LLM 请求失败、API Key 缺失时仅在流中展示文本，缺少全局 Toast/错误边界反馈。
7. **输入体验待优化**：单行输入框不支持换行，长文本输入不便。
8. **移动端细节**：键盘弹出时布局、底部安全区、输入框高度等需要进一步打磨。
9. **微信接入**：尚未开始。

## 3. 第一阶段：功能可用与 UI 完整

### 3.1 目标

让网页端所有核心功能真正可用，UI 各页面完整、状态有反馈；最重要的是，让智能体第一次像「AI 好友」一样能主动开口、保持角色一致性，为后续优化和微信接入打下体验基础。

### 3.2 关键任务

#### 3.2.1 修复模型提供商逻辑

- 在 `Agent` 模型中补充 `apiEndpoint` 字段，用于自定义模型地址。
- 根据 `modelProvider` 映射默认端点：
  - `openai` → `https://api.openai.com/v1/chat/completions`
  - `anthropic` → 通过 OpenAI 兼容端点或单独 Claude Messages API 适配
  - `deepseek` → `https://api.deepseek.com/chat/completions`
  - `custom` → 使用 `apiEndpoint`
- 修改 `api/routes/chat.ts` 中的 LLM 请求逻辑，按智能体配置动态选择端点。

#### 3.2.2 完善「AI 好友」对话核心交互

- **主动开口**：进入聊天页时，智能体自动发送一条符合其性格的「开场白」或问候，而不是等用户先说话。
- **角色一致性**：将 `persona` 强化为「性格卡」，包含角色背景、语气、口头禅、与用户关系；LLM 请求时严格注入，避免智能体「出戏」。
- **重新生成**：对最后一条 assistant 消息支持重新生成，可选项覆盖或追加。
- **清空对话**：在聊天页提供清空当前智能体历史记录的入口，后端支持按智能体删除消息。
- **消息重试**：发送失败或 LLM 流返回错误时，允许用户点击重试。
- **复制消息**：长按/点击菜单可复制单条消息内容。
- **编辑用户消息**：支持修改已发送消息并重新触发回复。
- **输入框升级**：将单行 input 改为可自适应高度的 textarea，支持 `Enter` 发送、`Shift+Enter` 换行。

#### 3.2.3 智能体创建辅助

- **预设模板**：提供「知心好友」「游戏搭子」「学习伙伴」「幽默损友」「温柔倾听者」等 AI 好友模板，一键填充性格、语气、开场白与模型参数。
- **表单校验**：名称必填、API Key 非空提示、温度/Max Tokens 范围校验。
- **头像裁剪/压缩**：上传前限制大小与格式，避免过大图片。

#### 3.2.4 基础记忆与关系

- **用户关键信息记忆**：在 `users` 表中增加 `memorySnapshot`（或独立 `memories` 表），让 LLM 在每次对话前获得「用户最近说过的重要信息」摘要，例如名字、喜好、最近事件。
- **简单关系进度**：记录与每个智能体的聊天轮数、最后互动时间，用于后续展示「认识第 N 天」「聊过 N 句」等关系感文案。
- **记忆更新机制**：在每次对话结束后，让 LLM 提取本次对话中值得记住的事实，异步更新记忆摘要。

#### 3.2.5 错误处理与状态反馈

- 引入全局轻量 Toast/Notice，用于：
  - 网络错误
  - API Key 未配置
  - 智能体保存成功/失败
  - 头像上传失败
- 为所有异步操作添加 loading 状态，防止重复提交。
- 为 `ErrorBoundary` 增加更友好的错误提示与刷新按钮。

#### 3.2.6 移动端基础适配

- 处理 iOS/Android 键盘弹出时的输入框位置（`visual viewport` 监听或 CSS `dvh`）。
- 底部导航适配安全区（`env(safe-area-inset-bottom)`）。
- 触控目标最小 44×44px。

#### 3.2.7 工程质量与 Render 部署

- `pnpm lint`、`pnpm check`、`pnpm build` 全部通过。
- 补充基础环境变量说明（`DATABASE_URL`、`NODE_ENV`、`PORT`）。
- Render 部署脚本可正常跑通迁移与构建；以 Render 作为第一阶段默认生产环境。

### 3.3 验收标准

- [ ] 选择 OpenAI/DeepSeek/自定义 API 后，聊天能正常收到流式回复。
- [ ] 进入聊天页时智能体会主动发送符合性格的开场白。
- [ ] 智能体在对话中能维持一致的性格与语气，不「出戏」。
- [ ] 对话页支持重新生成、清空、复制、重试。
- [ ] 创建智能体时可使用 AI 好友模板，保存后列表实时更新。
- [ ] 所有异步操作有 loading/错误反馈，无白屏或卡死。
- [ ] 主流手机浏览器（iOS Safari、Chrome Android）可正常聊天。
- [ ] `pnpm build` 无 TypeScript 与 Lint 错误。
- [ ] 成功部署到 Render 并通过 `/api/health` 健康检查。

## 4. 第二阶段：用户体验与 UI 优化

### 4.1 目标

在功能可用的基础上，把「AI 好友」感做深：让智能体更像一个持续陪伴、越来越懂用户的朋友，而不是用完即走的工具；同时保证在 Render 部署环境下的性能与稳定性。

### 4.2 关键任务

#### 4.2.1 视觉与动效

- **页面切换过渡**：路由切换时增加淡入/滑入动画，避免生硬跳转。
- **消息入场动画**：新消息以自然方式出现（淡入 + 轻微上移）。
- **卡片列表动画**：智能体列表加载、删除、新增时有 stagger 动效。
- **主题一致性**：确保所有页面、组件、弹窗都跟随 `theme-aurora` / `theme-colorful`，无硬编码颜色。
- **减少视觉疲劳**：提供柔和字体渲染、适当行高与消息气泡间距。

#### 4.2.2 AI 好友感深化

- **长期记忆**：引入 `Memory` 表，记录用户关键事实（喜欢的食物、最近在忙的事、生日等），每次对话前注入最新记忆摘要；支持在智能体设置中查看/删除记忆。
- **主动找话题**：当用户长时间未回复或每天首次打开时，智能体可根据记忆主动发起问候或分享「今天想聊的话题」。
- **角色状态/心情**：为智能体设计简单状态（开心、困倦、想念你），在聊天页顶部或头像旁以小状态呈现，影响其回复语气。
- **关系进度可视化**：展示「你们已经认识 X 天」「聊过 Y 句」，增强陪伴感。
- **语音消息（可选）**：支持用户发送语音，后端转文字后进入 LLM；assistant 回复可合成语音返回（TTS）。

#### 4.2.3 对话体验升级

- **Markdown 渲染**：assistant 消息支持代码块、列表、加粗、链接等常见格式，代码块可复制。
- **长文本优化**：超长回复自动折叠，支持展开；图片/链接预览。
- **引用与上下文**：用户可长按消息引用，被引用内容进入下一轮上下文。
- **快捷操作**：输入框上方提供常用快捷指令 / 历史快捷输入。
- **空状态升级**：根据场景给出不同插画与文案（无智能体、无对话、API Key 未配置等）。

#### 4.2.3 个性化

- **更多主题**：在极光、七彩基础上增加「暗夜」「清晨」等主题。
- **智能体开场白**：支持配置智能体首次对话时的欢迎语。
- **示例对话**：在智能体设置中可添加示例问答，帮助用户理解用法。

#### 4.2.4 触控与反馈

- 按钮、卡片、输入框增加 `active:scale` 与按压态。
- 发送/接收消息时有轻微震动反馈（移动端支持 `navigator.vibrate`）。
- 操作结果即时反馈（如删除智能体后列表动画移除）。

#### 4.2.5 性能与稳定性

- 长聊天记录虚拟滚动或分页加载，避免 DOM 过多导致卡顿。
- 图片懒加载与压缩。
- 前端路由、API 请求添加超时与重试。
- 减少不必要的全量重渲染。

#### 4.2.6 无障碍

- 为关键按钮添加 `aria-label`。
- 支持键盘导航（Tab、Enter、Esc）。
- 尊重 `prefers-reduced-motion`，关闭非必要动画。

### 4.3 验收标准

- [ ] 页面切换与消息发送有流畅动画，不卡顿。
- [ ] Markdown/代码块正常渲染，移动端可读。
- [ ] 至少提供 4 套主题，切换即时生效。
- [ ] 长聊天记录滚动流畅，1000 条消息无显著掉帧。
- [ ] 支持键盘完整操作主要流程。

## 5. 第三阶段：接入微信

### 5.1 目标

让用户无需打开网页，即可在微信内通过公众号或小程序与智能体进行多轮对话，且历史记录与网页端互通。

### 5.2 关键任务

#### 5.2.1 微信服务端接入

- **公众号/小程序服务器配置**：
  - 提供 URL、Token、EncodingAESKey 配置说明。
  - 实现微信服务器签名验证接口（`signature` + `timestamp` + `nonce`）。
- **新增微信路由**：`api/routes/wechat.ts`，处理：
  - GET 请求：服务器配置验证。
  - POST 请求：接收微信推送的消息与事件。

#### 5.2.2 用户映射

- 使用微信用户 `OpenID` 作为唯一标识。
- 首次微信访问时，创建一个对应的匿名 `User` 记录，建立 `wechatOpenId` 关联字段（需新增字段或独立表）。
- 网页端登录用户后续可绑定微信，实现多端历史互通（可选）。

#### 5.2.3 消息接收与回复

- 解析微信 XML 消息，提取：
  - 文本消息 `MsgType=text`
  - 语音消息 `MsgType=voice`（通过语音识别转文本）
  - 图片消息 `MsgType=image`（可描述或暂不处理）
  - 事件消息 `MsgType=event`（关注、取关、点击菜单）
- 根据用户当前绑定的「默认智能体」或关键词选择智能体。
- 复用 `api/routes/chat.ts` 中的 LLM 对话逻辑，但改为同步/异步两种回复模式：
  - 简单回复：5 秒内直接返回微信 XML 响应。
  - 复杂回复：先返回空响应，再通过微信「客服消息」接口异步推送结果。
- 用户发送的消息与 assistant 回复统一存入 `chat_messages`，网页端可见。

#### 5.2.4 智能体路由与入口

- 每个智能体可生成独立微信入口（菜单关键词、带参二维码、网页链接）。
- 支持通过关键词切换当前默认智能体，例如发送「切换 xxx」。
- 网页端可设置哪些智能体对微信开放。

#### 5.2.5 微信特色能力

- 关注公众号时发送欢迎语与智能体列表。
- 菜单点击直达常用智能体。
- 语音输入支持（需接入微信语音识别或自行 ASR）。
- 错误/超时消息以客服消息形式告知用户。

#### 5.2.6 安全与稳定性

- 验证微信请求签名，防止伪造。
- 消息幂等去重（按 `MsgId` 或自定义去重键）。
- 对微信 API 调用添加超时、重试、降级（回复固定提示语）。
- 增加微信消息队列或限流，避免高频消息打爆后端。

### 5.3 新增/调整数据模型

```prisma
model WechatBinding {
  id        Int      @id @default(autoincrement())
  openId    String   @unique
  userId    Int      @unique
  createdAt DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@map("wechat_bindings")
}
```

- 在 `users` 表中可补充 `defaultAgentId` 字段，表示微信默认对话智能体。

### 5.4 验收标准

- [ ] 微信公众号/小程序服务器配置验证通过。
- [ ] 用户在微信内发送文字，5 秒内收到智能体回复。
- [ ] 多轮对话上下文连续，与网页端历史互通。
- [ ] 可通过关键词或菜单切换智能体。
- [ ] 微信请求签名验证、消息去重、超时降级均生效。
- [ ] 生产环境能稳定处理并发消息。

## 7. 风险与依赖

| 风险 | 影响 | 应对 |
|---|---|---|
| 微信资质/公众号审核 | 第三阶段无法测试 | 提前注册并申请公众号/小程序，准备测试号 |
| 各家 LLM API 差异大 | 多模型支持困难 | 优先统一 OpenAI 兼容格式，特殊字段做适配层 |
| API Key 由用户自填 | 易出现余额/权限问题 | 错误提示要具体，引导用户检查 Key |
| 长聊天记录性能 | 第二阶段卡顿 | 提前规划分页/虚拟滚动 |
| 消息并发与超时 | 微信用户体验差 | 引入异步客服消息与降级提示 |

## 9. 结语

本开发文档以「先可用、再好用、再扩展」为节奏，优先保证核心对话链路稳定，再围绕移动端体验持续打磨，最后通过微信接入放大使用场景。每个阶段都有明确的验收标准，便于团队按里程碑推进与复盘。
